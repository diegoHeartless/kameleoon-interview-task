# kameleoon-interview-task

## Развертывание на GitHub Pages

Проект настроен для автоматического деплоя на GitHub Pages через GitHub Actions.

### Автоматический деплой

При каждом push в ветки `main` или `master` проект автоматически собирается и деплоится на GitHub Pages.

### Ручная настройка

Если при первом запуске workflow возникает ошибка "Get Pages site failed", выполните следующие шаги:

1. В настройках репозитория GitHub перейдите в **Settings** → **Pages**
2. В разделе **Source** выберите **GitHub Actions**
3. Если раздел **Source** недоступен или пуст, попробуйте:
   - Убедитесь, что у репозитория есть права на Pages (для public репозиториев доступно бесплатно)
   - Проверьте, что Actions включены в репозитории (**Settings** → **Actions** → **General** → убедитесь, что Actions включены)
4. После настройки Pages, workflow автоматически включит Pages при следующем запуске
5. После первого push в `main`/`master` ветку, workflow автоматически создаст деплой

### Локальный билд для GitHub Pages

Для локального тестирования билда с настройками GitHub Pages:

```bash
npm run build:gh-pages
```

### Данные

Проект автоматически использует данные:
- **В режиме разработки**: Пытается подключиться к локальному серверу (`http://localhost:3001/data`), если он запущен через `npm run server`
- **На GitHub Pages**: Использует статический JSON файл `public/data.json` (данные идентичны серверу)

Для локальной разработки с сервером:
```bash
npm run server  # Запустить сервер на порту 3001
npm run dev     # Запустить фронтенд
```

Для принудительного использования JSON файла (вместо сервера) установите переменную окружения:
```bash
VITE_USE_JSON_DATA=true npm run dev
```

### Примечание

Убедитесь, что название репозитория в `vite.config.ts` совпадает с реальным названием вашего репозитория на GitHub. Если репозиторий называется иначе, обновите строку `base` в `vite.config.ts`.
